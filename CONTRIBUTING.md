# Contributing to Jichi Insight

Jichi Insightは、政治的な主張を集める場所ではなく、公開資料を検証可能な形に整理するプロジェクトです。変更は、内容の面白さよりも根拠、再現性、中立性を優先します。

## Branch and pull request

- `main` への直接コミットは行いません。
- ブランチ名は `agent/<purpose>`、`feature/<purpose>`、`data/<area>`、`fix/<purpose>` を使用します。
- PRは小さく保ちます。コード、方法論変更、大量データ投入を可能な限り分離します。
- 原則としてSquash Mergeを使用します。
- PR本文に「変更内容」「理由」「影響」「検証」「出典」を記載します。

## Definition of done

変更は、該当する条件をすべて満たした時点で完了です。

### Code

- lint、typecheck、test、buildが成功
- エラー状態、空状態、モバイル表示を確認
- アクセシビリティ上の重大な問題がない
- 画面に出典・鮮度・品質を載せる設計を壊していない

### Data

- 一次資料URLと資料名がある
- 公開日または確認日がある
- ページ番号または該当箇所がある
- 金額の単位、年度、当初・補正・決算の区別がある
- 自動抽出と人手確認の状態が明示される
- 不明値を推測で埋めていない
- スキーマ検証が成功する

### Methodology

- 評価対象と評価主体の役割が明確
- 評価式、閾値、除外条件が説明される
- 既存評価への影響が確認される
- 恣意的な人物・政党別ルールがない
- 変更履歴が残る

## Source hierarchy

1. 自治体、議会、選挙管理委員会などの公式一次資料
2. 国の統計・公的データベース
3. 監査報告、議会議事録、契約・入札資料
4. 報道機関
5. 政党、候補者、議員の公式発信
6. その他の第三者情報

二次資料は調査の入口には使えますが、重要数値の確定根拠には原則として使いません。

## Data states

- `verified`: 一次資料と人手確認が完了
- `reviewed`: 一次資料に基づきレビュー済み
- `extracted`: 自動抽出済み、未レビュー
- `inferred`: 複数資料からの推定を含む
- `missing`: 必要情報を確認できない

公開評価に用いるのは原則として `verified` または `reviewed` です。

## Corrections

誤りを見つけた場合は、Data correction Issueを使用してください。対象URL、誤っている記述、正しい内容、根拠資料を記載してください。修正時は変更履歴を残します。
