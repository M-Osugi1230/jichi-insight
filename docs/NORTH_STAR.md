# North Star — Jichi Insightが存在する理由

## 1. 最上位の目的

Jichi Insightは、自治体や政治家を一方的に採点するためのサイトではありません。

> **住民が、自分自身で自治体、知事・市長、議会を評価できる情報環境をつくる。**

これが、対象地域、画面、技術、事業モデルが変わっても維持する最上位の目的です。

## 2. 出発点となった課題意識

自治体は予算、決算、事業評価、契約、監査、議会資料、選挙公報など、多くの情報を公開しています。

しかし、公開されていることと、住民が理解・比較・検証できることは同じではありません。

現在の主な問題は、情報の非公開だけではなく、次の分断です。

1. **資料の分断**
   - 一つの政策を追うために複数部署・複数サイト・PDFを横断する必要がある。
2. **資金と成果の分断**
   - いくら使ったかは分かっても、何を実行し、何が変わったかがつながっていない。
3. **公約と行政事業の分断**
   - 選挙時の約束が、どの予算・事業・KPIに対応するか確認しにくい。
4. **首長と議会の責任の混同**
   - 執行責任、審議責任、監視責任が区別されず、印象で評価されやすい。
5. **公開されていない情報の不可視性**
   - 未公開、未確認、該当なし、評価不能が区別されていない。
6. **単純ランキングの危険性**
   - 自治体規模、地域条件、権限、外部要因を無視した点数は住民を誤導する。

## 3. Jichi Insightがつなぐもの

```text
Promise        何を約束したか
   ↓
Money          いくら確保し、何に使ったか
   ↓
Action         何を実行し、誰と契約したか
   ↓
Result         目標に対して何が変わったか
   ↓
Accountability 誰が判断し、どう説明・監視したか
```

最終的には、次を一つの検証可能な流れとして表示します。

```text
税収・財源
  → 予算配分
  → 個別事業
  → 契約・支出
  → 実施内容
  → KPI・成果
  → 監査・議会審議
  → 首長の公約進捗
```

## 4. 評価対象と役割

### 自治体

財政健全性、行政効率、住民サービス、政策成果、将来持続性、情報公開を確認します。

### 知事・市長

公約の具体性、実行状況、関連予算、成果、方針変更の説明、組織統治を確認します。

### 議会

予算・条例の審議、修正、行政監視、政策提案、情報公開、住民参加を確認します。

首長と議会に同じ評価軸を適用しません。

## 5. 非交渉原則

1. 事実、比較、解釈、評価を分ける。
2. 重要な主張は一次資料へ接続する。
3. 不明・未公開・未確認を推測で埋めない。
4. 未公開、未確認、該当なし、評価不能を区別する。
5. 政策思想、政党、人物の好悪を採点しない。
6. 自治体、首長、議会を役割別に評価する。
7. 公約変更を自動的に失敗としない。理由、説明、代替策を見る。
8. 政策成果と行政・首長の因果関係を安易に断定しない。
9. 方法論、評価式、データ品質、変更履歴を公開する。
10. 訂正と反論の機会を提供し、重大な誤りを静かに消さない。
11. 評価に必要な証拠が不足する場合、無理に点数を出さない。
12. 対象数を増やすために、公開品質を下げない。

## 6. 初期パイロット

- 福岡県
- 福岡市
- 北九州市

最初に3自治体について、財政、重点事業、公約、議会、出典を縦に深くつなぎます。このモデルが成立してから地域を拡大します。

## 7. 優先順位

```text
信頼性
  > 理解しやすさ
  > データの深さ
  > 更新可能性
  > 機能数
  > 対象自治体数
```

## 8. 成功した状態

利用者が短時間で、次の問いに答えられる状態を目指します。

- この自治体はどのような財政状態か。
- 税金を主に何へ使っているか。
- 重要事業は何を目的にし、いくら使ったか。
- 目標に対して何が実現したか。
- 首長は何を約束し、どこまで進めたか。
- 議会は何を審議・修正・監視したか。
- その判断の根拠資料は何か。
- 何がまだ分からないか。

## 9. 変更ルール

この文書は、通常の機能追加や短期的な都合で変更しません。

変更する場合は、専用PR、ADR、影響分析、変更理由の公開を必要とします。
