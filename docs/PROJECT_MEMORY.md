# Project Memory — 着想と重要判断

## 着想

福岡県議会の海外活動費をめぐる報道をきっかけに、次の疑問からプロジェクトが始まりました。

- 税金を使った行為として正しかったのか。
- 宿泊、交通、日当、契約の費用は妥当だったのか。
- 視察や事業の目的と成果は何だったのか。
- 情報は公開されているのに、なぜ住民が判断しづらいのか。

調査すると、自治体には予算、決算、入札、監査、議会資料など多くの公開情報がありました。

一方で、資料は断片的で、一般の住民が一つの支出や事業を追跡し、費用対効果まで理解するのは難しい状態でした。

## 中核となる発見

> **公開されていることと、理解できることは異なる。**

Jichi Insightが解決する中心課題は、情報公開の有無だけではありません。

公開情報を、住民が利用できる知識へ変えることです。

## プロダクト仮説

自治体情報を次の単位でつなげると、住民が行政を事実ベースで理解できます。

```text
税収・財源
  → 予算
  → 個別事業
  → 契約・支出先
  → 実行内容
  → KPI・成果
  → 監査・議会審議
  → 首長のマニフェスト
```

## 長期的な評価対象

1. 自治体全体
2. 知事・市長
3. 議会
4. 個別議員
5. 選挙時の候補者・公約

ただし、対象を増やす前に自治体・首長・議会をつなぐデータモデルを完成させます。

## 重要な判断

- ブランド名は **Jichi Insight / 自治体インサイト**。
- サービス分類は自治体IR・行政アカウンタビリティ基盤。
- 初期対象は福岡県、福岡市、北九州市。
- アルファ版では人物の単一総合点を出さない。
- 自動抽出結果を、未確認のまま公開評価へ使わない。
- 首長と議会を同じ物差しで評価しない。
- 政策思想や政党の優劣を判定しない。
- 全国を浅く集める前に、一地域を縦に深く完成させる。
- 訂正・反論・変更履歴をプロダクトの中心機能とする。

## 開発時に毎回思い出す問い

- この変更は、住民が自分で判断するための根拠を増やすか。
- 出典から表示まで追跡できるか。
- 分からないことを、分かったように表示していないか。
- 首長、行政、議会の責任を混同していないか。
- 話題性やランキング性を、正確性より優先していないか。
- 対象数を増やすことで品質を下げていないか。

## 実行順序

1. North Starと方法論を固定する。
2. 公式資料の所在をカタログ化する。
3. 出典・品質・欠損を表示する共通UIを作る。
4. 財政データを3年度以上つなぐ。
5. 重点事業の予算から成果までつなぐ。
6. 首長公約を事業へ接続する。
7. 議会の審議・採決・支出・視察を接続する。
8. 訂正・反論手続と公開運用を検証する。
9. クローズドβを経て公開する。
